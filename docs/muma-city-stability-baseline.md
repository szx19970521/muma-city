# 牧马城市稳定性基线

更新时间：2026-06-22

这份文档先给当前项目定“稳定优先”的基调。它不是最终游戏设计方案，而是后续继续做角色、城市、智能体、交互之前必须遵守的工程边界。

## 当前状态

当前代码可以构建，但工作区已经积累了大量未提交改动，尤其集中在 Office3D、城市、角色、第一视角、启动保护、资产和测试上。继续直接往主场景里补丁式修改，容易出现以下问题：

- 旧 bug 没完全解决，新资产或新姿态又引入新 bug。
- 视觉对象、碰撞对象、导航对象不是同一份数据，导致“看得见但穿过去”或“看不见却有空气墙”。
- 第一视角手臂、玩家身体、智能体角色混用不同来源资产，坐标系、比例、动画根节点不一致。
- 3D 大组件过大，`CityBackdrop`、`OfficeShell`、`Office.tsx` 等文件已经承担过多职责。
- 打包后 renderer 主 chunk 偏大，后续继续直接塞模型和动画会增加白屏、卡顿和加载失败风险。

## 已验证基线

本机当前已通过：

- `npm.cmd test`
- `npm.cmd run typecheck:web`
- `npm.cmd run typecheck:node`
- Office3D / 第一视角 / 碰撞相关最小测试：
  - `Office3D.import.test.tsx`
  - `FirstPersonController.test.ts`
  - `FirstPersonAvatar.test.tsx`
  - `navigationCollision.test.ts`
- `npm.cmd run validate:first-person-rig`
- `npm.cmd run build:unpack`

需要注意：

- `validate:first-person-rig` 当前结果是通过，但主手臂资产处于 `visualReviewStatus: failed`，运行时回退为 `safe-placeholder-hands`。这表示“不会白屏”，不表示“手臂效果合格”。
- `model-discovery` 和 `connection-config-security` 原先依赖本机 loopback server。经本地验证，当前执行环境里 Node 进程连接自己启动的 `127.0.0.1` 测试服务器会 `ETIMEDOUT`。现在这类测试已改为 mock HTTP request 层，不再依赖本机端口。

## 稳定性红线

后续所有修改必须遵守：

1. 不再直接在主场景里试新角色资产、新手臂资产、新动画资产。
2. 角色、手臂、动画必须先进入角色实验室，通过截图/手动验收后才接入主场景。
3. 同一次任务只改一个子系统：角色、城市、碰撞、天气、交通、UI、启动稳定，不混改。
4. 视觉模型和碰撞数据必须共享来源。新增建筑、家具、门、树、车、路灯时，同步明确是否可穿行。
5. 自动门必须走统一的门状态机，不允许玩家、智能体、碰撞各自判断一套。
6. 新资产必须记录来源、许可、体积、运行时 fallback。加载失败只能降级显示，不能白屏。
7. 不把未通过视觉验收的资产设为主资产。
8. 任何会影响启动的改动，必须跑 `build:unpack`。

## 子系统边界

### 启动与白屏

目标：确保应用打不开时有诊断页，而不是白屏。

允许改动：

- Electron 主进程加载重试。
- renderer crash fallback。
- 打包资源路径。
- 启动测试。

不允许顺手改：

- Office3D 场景内容。
- 角色资产。
- 城市资产。

### 角色与动画

目标：先定玩家、智能体、NPC 的角色体系，再进入主场景。

玩家角色建议路线：

- 用 Quaternius 作为优先实验对象。
- 第一视角、第三视角、低头可见身体、V 切换必须作为同一个角色系统设计。
- 不再长期维护“独立手臂 + 另一个第三视角身体”的割裂方案，除非实验室验证能稳定对齐。

智能体建议路线：

- 先做到能看出工作、休息、对话、走路、开门。
- 工位办公需要手放到键盘/鼠标附近，但不在主场景里硬掰骨骼。

NPC 建议路线：

- 可使用简化动作集。
- 不需要真实智能体能力，只需要外观和行为自然。

### 城市与建筑

目标：建立统一城市生成规则，而不是每次手写一栋楼。

保留建筑：

- 公司主楼。
- 停车场。
- 小马餐厅。
- 小马车行。
- 已经明确设计过的功能建筑。

可调整区域：

- 自动生成的背景楼。
- 无功能的占位楼。
- 道路周边环境、公园、街区风格。

硬性要求：

- 新增建筑不得挡住已有招牌和入口。
- 建筑 footprint 必须进入碰撞系统。
- 远景高楼要有高度层级、窗户和夜间灯光，不允许大面积纯色灰块。

### 碰撞与空气墙

目标：把“能走哪里、不能走哪里”从视觉组件里抽出来。

必须统一：

- 玩家碰撞。
- 智能体导航。
- 建筑 footprint。
- 家具/车辆/树/灯杆/红绿灯碰撞。
- 门打开后的可通行区域。

空气墙处理原则：

- 看得见的实体可以挡路。
- 看不见的范围限制必须有明确边界和原因。
- 背景楼删除时，碰撞 footprint 必须同步删除。

### 性能与资源

当前风险点：

- renderer 主 chunk 已经超过 11MB。
- Office3D 文件数量和职责增长过快。
- 城市楼、NPC、车辆、雨、灯光、CanvasTexture 都可能堆叠成卡顿来源。

后续规则：

- 大资产按实验室和主场景分层加载。
- 背景楼优先使用 instancing / LOD / 稳定生成，不为每栋楼写独立复杂组件。
- 避免在 `useFrame` 中创建对象、贴图、材质、数组。
- CanvasTexture 只在文本变化时更新，不每帧更新。
- 新增天气、车辆、NPC 时必须说明数量上限。

## 推荐开发流程

每个任务按三路拆分，但最终由一个窗口整合：

- Subagent A：定位影响范围和已有代码。
- Subagent B：找测试、风险点和回归范围。
- Subagent C：给修复方案或手动验收清单。
- Main：只整合必要补丁，跑最小验证。

如果开多个 Codex 窗口，必须按文件所有权分工：

- 角色实验室窗口：只改 `office3d/avatarLab`、角色资产索引、实验室 UI。
- 城市窗口：只改城市生成、建筑外观、公园、背景楼。
- 碰撞窗口：只改 `navigationCollision`、footprint、门/传感器。
- 应用 UI 窗口：只改非 3D 的 React UI。
- 主协调窗口：负责合并、跑测试、打包。

不要让两个窗口同时改 `Office3D.tsx`、`Office.tsx`、`CityBackdrop.tsx`、`OfficeShell.tsx` 这类核心文件。必须改时，先由主协调窗口分出接口。

## 第一阶段建议

稳定第一阶段不继续做新视觉功能，先完成：

1. 建立 `validate:game` 聚合脚本，至少包含 typecheck、Office3D import、碰撞、第一视角、rig validator。
2. 把当前第一视角手臂标记为实验失败状态，主场景只保留稳定占位，不再继续临时摆姿势。
3. 角色实验室继续做 Quaternius / Rocketbox / 其他资产筛选，不通过验收不接主场景。
4. 为城市建筑和碰撞建立统一 footprint 数据，之后再处理空气墙和穿模。
5. 对 renderer 主 chunk 和 3D 资产做体积预算，避免继续直接堆进主包。

## 手动验收标准

任何进入主场景的 3D 改动至少要看：

- 启动是否白屏。
- 第一视角和第三视角是否都能进。
- 玩家是否能正常走、跑、跳、进出门。
- 门是否自动开关且不穿模。
- 新增实体是否有合理碰撞。
- 夜晚、雨天、白天是否都能看清。
- 主场景是否明显卡顿。

没有经过这些检查的视觉改动，只能留在实验室或功能分支。
